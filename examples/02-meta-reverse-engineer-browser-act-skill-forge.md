# 示例 02 — Meta Demo：反向推导 browser-act/skill-forge

> **本示例的目的**：用 skillforge 七步法**反向分析**一个已有的、运行良好的 Skill（`browser-act-skill-forge`），展示 skillforge 不仅能**设计**新 Skill，还能**理解/评审**别人的 Skill。
>
> **为什么是 browser-act/skill-forge**：
> - 1950+ stars 的活跃项目（借势）
> - 跟开源/AI Agent 圈读者高度相关
> - 文档完整（SKILL.md + 3 个 references），推导有充足素材
> - 复杂度高（Phase 0→1→2→3→Delivery），能撑起七步

---

## 0. 这个 Skill 在做什么（一句话）

```
输入：  用户意图（"我要从 X 网站抓数据"或"做一个跟 Y 一样的事"）
输出：  可重复使用的 Skill 包（SKILL.md + Python 脚本）
核心：  探站一次，永久复用 —— 后续 300 个关键词走同一条路径
位置：  https://github.com/browser-act/skills/tree/main/browser-act-skill-forge
```

## 1. 我们的任务

用 skillforge 七步法拆解它，**回答两个问题**：
1. 它为什么能拿到 1950 stars？（设计上的优秀之处）
2. 它可以怎么演进？（skillforge 给的扩展预设）

---

## 七步反向推导

### 第零步：搜（这个 Skill 的"母体"是什么）

**问题**：在它之前，这类"从意图生成 Skill"的工具存在吗？

**搜出来的东西**：
- **Coze** / **Dify** 的工作流模板（半自动，需要人拖拽）
- **GPT Maker** / **V0** 类工具（生成 UI，不生成 Skill）
- **LangChain Hub**（人工分享的 prompt 模板）
- **probot**（GitHub 自动化机器人，但代码固定）

**学到的关键点**：
- **API 优先 → DOM 兜底**是行业最佳实践（避免脆弱的 DOM 选择器）
- **Explore once, reuse forever** 是所有抓取工具的圣杯
- **零配置部署**是用户友好的关键（不要让用户注册 5 个账号）
- **Sub-Agent 自动化测试**是 AI Agent 圈新范式

**Skill 自己的"搜"的产出**（from its SKILL.md）：
- 工具：browser-act / 自动化测试 / Python 脚本生成
- 流程：Phase 0 → 1 → [2 → 3 loop] → Delivery
- 踩坑：API 限流 / DOM 变化 / 登录态丢失
- 阈值：探索 100 步上限

---

### 第一步：分（阴阳切清楚）

**阳面（它要做什么）—— 7 条**：
1. 接受用户的自然语言意图
2. 解析意图：目标站点 + 业务目标 + 执行意图（立即跑 vs 造 Skill 后用）
3. 站点研究（没有 URL 时）
4. 探查能力（API → DOM 兜底）
5. 生成可复用的 Skill 包（SKILL.md + scripts/）
6. 自动化测试 + 合规自检
7. 安装生成的 Skill，按需批量执行

**阴面（它绝不做什么）—— 8 条**：
1. ✗ 不替用户注册开发者账号 / 拿 API key
2. ✗ 不绕过权限/付费墙（账号限制 ≠ 技术失败）
3. ✗ 不调用第三方抓取服务
4. ✗ 不调用官方开放 API（要零配置）
5. ✗ 不修改页面内容
6. ✗ 不持久化用户登录态
7. ✗ 失败不静默吞掉 —— 必须告诉用户哪些没覆盖
8. ✗ 探索不超过 100 步（防爆栈 / 防 AI 跑飞）

**8 条禁止规则的威力**：这是它能"零配置部署"+"不惹版权麻烦"+"用户敢用"的根。任何一条破例，信任体系就崩。

---

### 第二步：叠（四力）

**太阳（工具）—— 5 个核心**：
- **T1. browser-act**（核心浏览器，自带隐身/反爬）
- **T2. analyzer**（自然语言意图解析）
- **T3. explorer**（API/Network/DOM 三路探索）
- **T4. generator**（SKILL.md + Python 脚本生成）
- **T5. verifier**（Sub-Agent 测试 + 合规自检）

**5 个工具，刚好落在 skillforge 推荐的 3-10 个范围内**。

**少阳（任务顺序）—— 5 个阶段 + 1 个 loop**：
```
Phase 0: 工具检测（一次）
Phase 1: 需求分析 + 站点研究 + 计划确认（一次）
[Loop]   Phase 2 (探查) → Phase 3 (生成) — 每个 capability 跑一遍
Delivery: 自动化测试 → 安装 → 报告 → 执行
```

**关键设计**：把"探查"和"生成"绑成 loop，每个 capability 独立闭环。这样：
- 失败的 capability 不影响其他
- 增量更新某个能力时不用重跑全部
- 用户的"看效果"反馈循环短

**少阴（判断）—— 6 个，都带量化阈值**：
- 判断 1：是否已存在 Skill? → 跳过构建
- 判断 2：API 可用? → 是 → API 路径；否 → DOM 路径
- 判断 3：失败类型？→ 确定性（1 次）/ 瞬时（重试 1 次）
- 判断 4：探索进度？→ ≤ 100 步
- 判断 5：生成脚本能跑通？→ eval 验证 + 异常场景
- 判断 6：权限限制？→ 标"Known Limitations"，不算失败

**太阴（门控）—— 5 条**：
- □ Phase 3d 合规自检（必须读文件验证，不能心理断言）
- □ 测试用 Sub-Agent（不污染主 session 上下文）
- □ 输出格式按 template
- □ 文件路径约定（output/{skill-name}/...）
- □ 失败必须告知（"哪些没覆盖"）

### 第三步：展（八卦完整度 8/8）

| 八卦 | 它的填法 | 评价 |
|------|----------|------|
| 乾（输入）| 用户自然语言意图（含可选 URL）| ✓ 简单清晰 |
| 坤（输出）| output/{skill-name}/{site-slug}-{capability-slug}/{SKILL.md, scripts/*.py}| ✓ 路径约定明确 |
| 震（起势）| Phase 0 工具检测 | ✓ 单一起点 |

| 离（标准）| 能稳定获取/触发，方法可重现 | ✓ 简洁 |

**八卦全填，无悬空维度。** 这是它能形成"可重复使用"产物的前提。

### 第四步：衡（五行传导链）

**木（方向）**：
- 方向：降低"做一个新站点爬虫"的边际成本到接近 0
- 推动力（水）：下游明确 = 用户的批量任务
- 判断力（火）充沛：6 个判断都有量化阈值
- 金克木：8 条禁止规则约束方向（避免"啥都做"的失控）

**火（判断）**：
- 来自木：方向清晰 → 知道要判断什么
- 产生土：判断 → 生成 SKILL.md + 脚本
- 水克火：失败分类（确定性/瞬时）= 给硬判断加缓冲

**土（沉淀）**：
- 沉淀是：output/{skill-name}/ 下的完整 Skill 包
- 变结构：沉淀变成可重用的 Skill 库（金）
- 木克土：每次重新探查允许打破旧的沉淀（避免复用死板的旧结构）

**金（结构）**：
- 工具边界：5 个工具 / 8 条禁止 / Phase 划分
- 金生水：结构稳定 → 上下游协议（输出格式）可连通
- 火克金：阈值可调（100 步、1 次重试），不被僵化

**水（流动）**：
- 上游：用户意图
- 下游：生成的 Skill → 用户的批量执行
- 水生木：流动支撑新方向（每个新站点 = 新方向）
- 土克水：沉淀过多会阻塞（这就是为什么 100 步上限）

**五行完整闭环。** 没有任何一个力量单独过弱或过旺。

### 第五步：验（失衡检查 + 扩展性）

**失衡检查 —— 8 个症状全过**：

| 失衡症状 | 出现？ | 备注 |
|----------|--------|------|
| 想做太多但做不了 | ✗ | 阳面 7 条都有对应工具和判断 |
| 一直在做但没产出 | ✗ | 输出固定（SKILL.md + scripts/）|
| 有产出但很散乱 | ✗ | 3 个 reference 文件 + template 强制规范 |
| 输出对了但上下游不通 | ✗ | 输出格式 + 路径约定明确 |
| 上下游通了但没有方向 | ✓ 检查 | 扩展方向需明确定义（见下）|
| 一直在判断但不能决策 | ✗ | 6 个判断都是 yes/no，决策清晰 |
| 结构过度僵化无法调整 | ✗ | 阈值可调 |
| 方向过度发散无法执行 | ✗ | 7 条阳面都是核心场景 |

**扩展性验证**：
- 方向已定义：✓ 跨站点 / 跨能力复用
- 扩展方向：
  - 轻量扩：增加"浏览器指纹"探测（不同浏览器效果不同）
  - 重量扩：支持增量更新（Skill 已存在时只更新变更的 capability）
  - 架构扩：Skill 市场（用户分享自己生成的 Skill）
- 扩展门控：
  - 新能力不破 8 条禁止
  - 增量更新不破原 Skill 的稳定性
  - Skill 市场不破零配置原则

**五行生克顺畅，无失衡。** → 验通过。

## 它为什么能拿 1950 stars（设计上的 5 个亮点）

从 skillforge 反推的视角看，它的成功不是偶然，是 5 个设计选择都做对了：

### 亮点 1：API 优先 → DOM 兜底
- 大多数竞品上来就 DOM 选择器，**网站一改版就废**
- 它先抓网络包找 API，**找 API 失败才用 DOM**
- 这条原则让生成的 Skill 寿命长 10 倍

### 亮点 2：8 条禁止规则护城河
- 不是技术护城河，是**信任护城河**
- "不绕过权限 / 不调第三方 / 零配置" = 用户敢用、企业能部署
- 这让 BrowserAct 公司能卖付费层（隐身浏览器+代理）而不被告

### 亮点 3：Phase 0→1→[2→3 loop]→Delivery
- 不是一锅炖，**职责分离清晰**
- Phase 2/3 绑成 loop = 失败隔离 + 增量更新友好
- 这是 skillforge 「叠」（少阳任务链）做对的典范

### 亮点 4：Sub-Agent 测试 + 合规自检
- 用 Sub-Agent 测 = 不污染主 session 上下文
- "心理断言不算数" 这条 = 强制可验证
- 这两条让生成的 Skill 真的能跑，不是看起来像

### 亮点 5：失败分类（确定性/瞬时）
- 这是**反 AI 蛮干**的精妙设计
- AI 容易"重试到天荒地老"，它强制 1 次就换路
- 100 步上限 = 防失控

## 它可以怎么演进（skillforge 给的扩展预设）

按"轻量扩 → 重量扩 → 架构扩"：

### 轻量扩（只填八卦，不动结构）

- **支持视频/音频抓取**（不只是 HTML）
- **支持 OAuth 站点**（需要登录态，但定义清楚什么算"已登录"）
- **支持 GraphQL**（专门处理 query 结构）

### 重量扩（回到叠，加能力）

- **增量更新**：Skill 已存在时，只更新变更的 capability（不重跑全部）
- **多站点并发**：一次任务跨 5 个相似站点（如竞品监控）
- **Skill 评分系统**：自动给生成的 Skill 打分（稳定性/复用度/失败率）

### 架构扩（回到衡，加新维度）

- **Skill 市场**：用户分享自己生成的 Skill，形成生态
- **Skill 链编排**：自动把多个 Skill 串成 pipeline（fetch → clean → analyze）
- **Skill 版本管理**：Skill 升级时保留旧版本（防 API 改了，新版还没适配）

## 用 skillforge 反推别人的 Skill（一种新技能）

**这个 demo 教的事**：

- 不是 skillforge 只能用来**设计**新 Skill
- 它也能用来**理解**别人的 Skill（"为什么这个好"）
- 还能用来**评审**已有的 Skill（"哪里可以改进"）

**这三种用法**：
| 用法 | 适合场景 | 输出 |
|------|----------|------|
| **设计** | 想到一个意图 | 新的 SKILL.md |
| **理解** | 看别人的 Skill | "它为什么好"的分析 |
| **评审** | 自己写完不放心 | 失衡检查 + 改进建议 |

**这一招对学习者特别有用**：
- 看到好的开源 Skill → 用 skillforge 反推 → 学到设计模式
- 看到烂的 Skill → 用 skillforge 反推 → 看到它哪里失衡

## 自检：本次反推有没有问题

实事求是：
- ✗ 没真跑过这个 Skill 的代码（基于 SKILL.md 文档推导）
- ✗ references/ 下的 3 个文件没读
- ✗ 不知道 5 个工具的内部实现
- ✗ 不知道它实际的失败率是多少

**反推的局限**：skillforge 反推**只能从公开文档/可见行为**看，**看不到实现细节的 bug**。要补这一层，得跑过 + 读过代码。
