# 示例 01 — GitHub Issue 自动分流与回复草稿

> **本示例的目的**：演示如何用 skillforge 七步法，从一句"我想自动处理 GitHub Issue"推导出完整 Skill。
>
> **跟 PRINCIPLES.md 区别**：那边有"竞品价格监控"实例，本文换一个**跟开源项目维护直接相关**的场景，展示方法论在不同领域的可复用性。

---

## 0. 意图与读者

**意图**（用户的原话）：
> "我做开源项目，每天被 Issue 淹没。能不能做一个 Skill，自动把 Issue 分流、起草回复，让我只处理真正需要人判断的部分？"

**最终目标读者**：
- 中小规模开源项目维护者
- 每天 5-50 个新 Issue
- 希望 AI 处理 80% 重复性工作

**本示例产出**：
1. 一个完整的推导过程（搜→分→叠→展→衡→验→成）
2. 一个可直接复制使用的 `SKILL.md` 草稿
3. 自检清单 + 扩展预设

---

## 1. 第零步：搜（搜现有 Skill）

**问什么**：网上有没有做类似事情的 Skill？

### 搜什么

```
□ GitHub: 搜 "github issue triage bot" / "issue auto label"
□ Hugging Face: 搜 "issue classification" / "intent classification"
□ Coze / Dify: 搜 "客服工单" "自动分类"
□ 关键词组合: "Claude Skill" + "github" / "issue management"
```

### 搜到的关键信息

**工具选型参考**：
- 传统做法：`probot` + GitHub Actions + 关键词正则
- LLM 时代：直接用 LLM 读 issue 内容分类，比正则准 30%
- 典型组合：GitHub Webhook → LLM 分类器 → 自动打 label

**流程参考**：
```
新 Issue → 拉取 → 分类 → 打 label → 草拟回复 → 等维护者确认 → 发送
   ↑                                         ↓
   └────────── 维护者修改后回流 ←─────────────┘
```

**踩过的坑**：
- 关键词分类在英文 Issue 准确率只有 60%
- 中文 Issue 几乎没有现成训练数据
- "草拟回复" 容易被滥用成"自动回复" → 维护者失去控制感
- 训练数据有偏差：把"提问"误分类为"bug"

**行业默认阈值**：
- 自动打 label 的置信度阈值：≥ 0.85
- 草拟回复的相似度阈值：≥ 0.7（避免答非所问）
- 升级人工的触发条件：涉及 breaking change / 安全 / 金钱

### 搜的产出

- 工具候选清单：`github-api`（拉 issue）+ `LLM`（分类/草拟）+ `github-api`（打 label）+ `notification`（通知）
- 流程参考：分类 → 草拟 → 确认 → 发送
- 异常处理参考：拉取失败、API 限流、LLM 幻觉
- 质量门控参考：置信度阈值、升级机制

## 2. 第一步：分（切阴阳）

**问什么**：AI 拿到这个意图，最容易做什么过头？数据/决策属于谁？什么必须留给人？

### 问 AI 最容易做过头的事

```
问：AI 拿到 Issue 后最可能做过头什么？

答1：自动回复 Issue（而不是草拟）
  → 维护者失去对项目话语的控制
  → ✗ 禁止：自动发送任何对外回复

答2：误关闭 Issue（看到"问题已修复"就关）
  → 用户被冷处理，体验崩坏
  → ✗ 禁止：自动关闭任何 Issue

答3：过度分类（把每个 Issue 都打 5 个 label）
  → label 失去筛选意义
  → ✗ 禁止：单个 Issue 最多 2 个 label
```

### 问数据/决策属于谁

```
问：Issue 标签的定义权属于谁？
答：项目维护者
  → ✗ 禁止：创建项目维护者没定义的 label

问：代码相关的判断属于谁？
答：开发者（issue 提交者或维护者）
  → ✗ 禁止：AI 替开发者承诺修复时间
```

### 问什么必须留给人类

```
问：什么判断必须人来？
答1：是否合并 PR（涉及代码质量）
  → ✗ 禁止：AI 替人合并 PR

答2：是否升级为"安全 issue"
  → ✗ 禁止：AI 自定安全等级

答3：是否 ban 用户
  → ✗ 禁止：AI 对用户做出惩罚性动作
```

### 阴阳汇总

**阳面（要做什么）**：
1. 拉取新 Issue
2. 分类（bug/feature/question/docs/duplicate）
3. 打 label（不超过 2 个）
4. 评估优先级（紧急/正常/低）
5. 草拟回复（不发送）
6. 高置信度的归档候选（仅打 label，不关闭）
7. 通知维护者有新待审

**阴面（不能做什么）**：
1. ✗ 不自动发送对外回复
2. ✗ 不自动关闭 Issue
3. ✗ 不创建项目维护者没定义的 label
4. ✗ 不替开发者承诺修复时间
5. ✗ 不替人合并 PR
6. ✗ 不自定安全等级
7. ✗ 不对用户做惩罚性动作
8. ✗ 不跨仓库操作

## 3. 第二步：叠（阴阳叠四力）

**问什么**：把阳面拆成工具、任务、判断、门控。

### 太阳（工具）

```
阳面 1：拉取 Issue        → github-api (read)
阳面 2：分类              → LLM (classifier)
阳面 3：打 label          → github-api (write)
阳面 4：评估优先级        → LLM (priority)
阳面 5：草拟回复          → LLM (draft)
阳面 6：高置信度归档候选  → LLM (confidence)
阳面 7：通知维护者        → notification (feishu/slack/email)
```

合并同类：
- `github-api` 拆为 `github-read` 和 `github-write`（读和写权限分离，安全）
- 三个 LLM 调用合并为一个 `llm-analyze` 工具（一次调用出三个结果，省钱）
- `notification` 保持独立

**最终 4 个工具**：
- T1. `github-read` — 拉取 Issue 列表和详情
- T2. `llm-analyze` — 分类/优先级/草拟回复
- T3. `github-write` — 打 label
- T4. `notify` — 通知维护者

### 少阳（任务顺序）

```
Step 1: github-read 拉取新 Issue
Step 2: llm-analyze 同时产出 [category, priority, draft_reply, confidence]
Step 3: 判断置信度（见少阴）
  └─ 高置信度 → 走 Step 4
  └─ 低置信度 → 跳过 Step 4，走 Step 5
Step 4: github-write 打 label
Step 5: notify 通知维护者（含草拟回复和置信度）
Step 6: 维护者手动 review 后决定是否回复/关闭
```

### 少阴（判断）

```
判断 1: 拉取成功了吗？
  → 失败：停止任务，发告警给维护者（"GitHub API 异常"）
  → 成功：进入 Step 2

判断 2: LLM 分类置信度 >= 0.85 吗？
  → 失败：跳过打 label，只通知（让人来分类）
  → 成功：进入 Step 3

判断 3: 草拟回复的相似度 >= 0.7 吗？
  → 失败：在草拟回复顶部加 "⚠️ AI 信心不足，请人工重写" 标记
  → 成功：正常提交草拟

判断 4: 是否涉及以下关键词？["security", "vulnerability", "CVE", "隐私", "数据泄露"]
  → 命中：升级为"高优先级"，且额外通知所有维护者（不能只通知一人）
  → 未命中：按正常优先级通知

判断 5: 草拟回复中是否承诺了时间/版本？
  → 是：删除该承诺，替换为"维护者会尽快回复"
  → 否：直接提交
```

### 太阴（门控）

```
□ 每日处理的 Issue 数有日志
□ 草拟回复的置信度分布有统计
□ label 应用记录可追溯
□ 高敏感关键词命中 100% 升级到所有维护者
□ 自动 label 不超过 2 个/Issue
```

## 4. 第三步：展（四力展八卦）

**问什么**：把四力展开成输入/输出/起势/流向/异常/标准/边界/握手。

### 乾（输入）

```
- GitHub repo 坐标：owner/repo
- 维护者自定义 label 列表：{label_name: color}
- LLM prompt 模板：分类/优先级/草拟 三个模板
- 通知渠道：飞书/Slack/email webhook
- 历史 Issue 数据：用于校准 LLM 分类（可选）
```

### 坤（输出）

```
- daily_issue_triage.json: 当日处理的 Issue 摘要
  {
    "date": "2026-06-06",
    "processed": 23,
    "auto_labeled": 18,
    "manual_needed": 5,
    "high_priority": 2,
    "sensitive_keywords_hit": 1
  }
- triage_log.md: 每个 Issue 的处理记录
  - Issue#42: bug, high, 草拟回复 "请提供复现步骤"
  - Issue#43: question, normal, 草拟回复 "请参考 docs/x.md"
- GitHub Issue 本身的 label（实际写入）
- 飞书/Slack 消息（实际推送）
```

### 震（起势）

```
触发方式（三选一）：
- Cron: 每天 09:00 跑一次
- Webhook: 新 Issue 进来 5 分钟内触发
- 手动: 维护者说"处理一下积压 Issue"
```

### 巽（流向）

```
- 下游 1: 维护者本人（接收通知 + review 草拟）
- 下游 2: 项目仪表盘（消费 daily_issue_triage.json）
- 下游 3: 周报生成（消费 triage_log.md，统计趋势）
```

### 坎（异常）

```
- GitHub API 限流 → 暂停 10 分钟，按指数退避重试
- LLM 调用失败 → 用上次的缓存结果，标注"未及时更新"
- LLM 输出 JSON 格式错 → 重试 1 次，失败则降级为纯通知
- 草拟回复中含 "merge" / "close" / "fixed" 等动作词 → 替换为占位符
- 网络断 → 本地缓存待处理列表，恢复后批量跑
```

### 离（标准）

```
- 每日 09:00 前完成昨日所有新 Issue 的 triage
- 自动打 label 的准确率（人工抽检）>= 90%
- 草拟回复的人工修改率 <= 30%（即 70% 草拟可直接用）
- 敏感关键词命中 100% 升级
- 处理 100 个 Issue 总耗时 <= 30 分钟
```

### 艮（边界）

```
- 不自动发送对外回复
- 不自动关闭 Issue
- 不创建项目维护者没定义的 label
- 不替开发者承诺修复时间
- 不替人合并 PR
- 不自定安全等级
- 不对用户做惩罚性动作
- 不跨仓库操作
- label 数量 <= 2 个/Issue
```

### 兑（握手）

```
- 输出文件路径约定：./output/issue-triage/{date}/
- 日报 JSON: daily_issue_triage.json
- 处理日志: triage_log.md
- 通知 payload: {issue_url, category, priority, draft_reply, confidence, sensitive_flag}
- 上游/下游约定: 本 Skill 是独立的，下游通过读文件消费
```

## 8. 最终产出：可复制的 SKILL.md 草稿

```markdown
---
name: github-issue-triage
description: 自动分流 GitHub Issue、评估优先级、草拟回复（不发送），让维护者只处理需要判断的部分。
---

# GitHub Issue Auto-Triage

## 触发条件
- 定时：每天 09:00
- 事件：新 Issue 进来 5 分钟内
- 手动：用户说"处理 Issue 积压"

## 输入
- GitHub repo 坐标
- 项目维护者定义的 label 列表
- LLM prompt 模板（分类/优先级/草拟）
- 通知渠道配置

## 输出
- GitHub Issue 自动打 label（≤ 2 个）
- daily_issue_triage.json
- triage_log.md
- 飞书/Slack 通知

## 工具
- github-read
- llm-analyze
- github-write
- notify

## 任务
1. 拉取新 Issue
2. LLM 分析（分类 + 优先级 + 草拟回复 + 置信度）
3. 置信度判断 → 决定是否打 label
4. 打 label
5. 通知维护者（含草拟回复和置信度）
6. 维护者手动 review

## 判断（量化）
- 拉取成功
- 分类置信度 ≥ 0.85
- 草拟相似度 ≥ 0.7
- 敏感词命中 → 升级所有维护者
- 草拟中含时间/版本承诺 → 替换为占位符

## 禁止（8 条）
- ✗ 不自动发送对外回复
- ✗ 不自动关闭 Issue
- ✗ 不创建项目维护者没定义的 label
- ✗ 不替开发者承诺修复时间
- ✗ 不替人合并 PR
- ✗ 不自定安全等级
- ✗ 不对用户做惩罚性动作
- ✗ label ≤ 2 个/Issue

## 异常
- GitHub API 限流 → 指数退避
- LLM 失败 → 用上次缓存
- LLM 输出格式错 → 重试 1 次
- 草拟含动作词 → 替换占位符
- 网络断 → 本地缓存待处理

## 标准（门控）
- 每日 09:00 前完成
- 自动 label 准确率 ≥ 90%（抽检）
- 草拟人工修改率 ≤ 30%
- 敏感词命中 100% 升级
- 100 Issue 总耗时 ≤ 30 min

## 扩展预设
- 轻量扩：多通知渠道
- 重量扩：自动 merge 友好 PR
- 架构扩：多仓库并行

## 扩展门控
- 新渠道不破通知 payload
- 新能力不破 8 条禁止
- 多仓不破数据隔离
```

## 10. 一句话回顾

**意图**（1 行）：
