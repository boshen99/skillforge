# 配方 — 五种场景的快捷推导入口

> **用法**：选一个最接近的场景 → 按该入口的引导走七步 → 完成
> **核心**：每个入口带特定问题，帮助你在该场景下更快完成"搜/分/叠/展/衡/验/成"
> **注意**：入口不同，但七步元认知不变，只是每步的问题更有针对性

---

## 入口选择

```
┌────────────────────────────────────────────────┐
│ 我的 Skill...                                   │
│                                                │
│ ├─ 是某个链路的一环（采集/清洗/分析/洞察）     │
│ │      → 入口一：链路节点                      │
│                                                │
│ ├─ 不属于任何链路，有自己的上下游              │
│ │      → 入口二：独立领域                      │
│                                                │
│ ├─ 是某个链路的一环（理解/改变/守住）          │
│ │      → 入口三：战略链路                      │
│                                                │
│ ├─ 核心是统计（是什么）+洞察（为什么）         │
│ │      → 入口四：分析洞察                      │
│                                                │
│ └─ 就是想快，简单做一个                        │
│        → 入口五：5分钟速成                      │
└────────────────────────────────────────────────┘
```

## 入口一：链路节点

### 适用场景

```
你的 Skill 属于：
  VOC链路：data-collection → data-etl → pattern-finder → voc-insight
  或：在某个节点上扩展（加一个子节点）

判断：它有明确的上游和下游吗？
  有 → 链路节点 ✓
  没有 → 可能是独立 Skill → 跳入口二
```

### 链路节点特有的"分"问题

```
除了通用三问，还要问：

问题L1：这个 Skill 的上游会给我什么数据？
       → 乾（输入）的大部分内容从这里来

问题L2：这个 Skill 要给下游什么数据？
       → 坤（输出）的主要内容从这里来

问题L3：上下游最容易误以为我也能做什么？
       → 艮（边界）的关键禁止从这里长出来

问题L4：上下游的数据格式是什么？
       → 巽（流向）握手协议从这里定义
```

### 链路节点特有的"叠"问题

```
太阳（工具）推导时，问：
  上游用什么工具？→ 我需要兼容吗？
  下游需要什么工具？→ 我需要输出它能读的格式吗？

少阳（任务）推导时，问：
  上下游的 Step 对齐了吗？→ 我的 Step 1 对应上游哪个输出？
                          → 我的最后一步对应下游哪个输入？

坎（异常）推导时，问：
  上游数据出问题了怎么办？→ 我要做容错还是拒绝处理？
  下游接收不了怎么办？→ 我要等待还是告警？
```

## 入口二：独立领域

### 适用场景

```
你的 Skill：
  □ 不属于 VOC 链路
  □ 不属于品牌战略链路
  □ 有自己独立的上下游（或完全没有上下游
  □ 不和其他 Skill 共享数据格式

示例：内部知识库、法律合规检查、产品参数库、竞品监控
```

### 独立 Skill 特有的"分"问题

```
问题I1：用户最担心 AI 做什么？
       → 这是第一条禁止

问题I2：这个领域的专业判断是什么？
       → 这决定少阴（判断）的深度

问题I3：出了错谁来负责？
       → 这决定太阴（门控）的严格程度

问题I4：这个领域有没有现成的规范/标准？
       → 这决定离（标准）怎么填
```

### 独立 Skill 特有的"展"问题

```
巽（流向）：无固定下游
  → 定义：这个 Skill 完成后谁会读？
  → 可能是：人类用户、外部系统、定时触发器

兑（握手）：
  → 没有标准格式，定义自己的握手协议
  → 核心三要素：输出什么 + 交给谁 + 怎么触发
```

## 入口三：战略链路

### 适用场景

```
你的 Skill 属于：
  L1 理解分类 — 我在哪里？（用户洞察/市场分析/赛道评估）
  L2 改变分类 — 我要去哪？（重构/重新定位/验证假设）
  L3 守住分类 — 我怎么守住？（品牌建立/执行落地）

判断：你是在做决策还是在分析数据？
  决策驱动 → 战略链路 ✓
  数据驱动 → 可能不是战略链路，跳入口一/四
```

### 战略链路特有的"分"问题

```
问题S1：这个层级（L1/L2/L3）不能跳级做什么？
       L1：不能给执行建议（还在理解阶段）
       L2：不能跳过验证直接给结论
       L3：不能给战略选项（已在执行阶段）

问题S2：什么决策必须留给人类？
       战略方向/资源分配/生死抉择

问题S3：你的假设能被证伪吗？
       能 → 可以进入L2  不能 → 还在L1
```

### 战略链路特有的"叠"问题

```
L1（理解分类）的四力：
  太阳：访谈工具、分析工具、画图工具
  少阳：收集洞察 → 分类归纳 → 形成理解
  少阴：用户真的这么想吗？证据在哪？有没有反例？
  太阴：□ 有数据支撑  □ 有分类  □ 有优先级

L2（改变分类）的四力：
  太阳：假设工具、验证工具、决策工具
  少阳：提出假设 → 设计验证 → 确认方向
  少阴：假设够大胆吗？能证伪吗？证据够吗？
  太阴：□ 假设清晰  □ 有验证路径  □ 结论可行动

L3（守住分类）的四力：
  太阳：执行工具、监控工具、评估工具
  少阳：制定计划 → 分配资源 → 执行落地
  少阴：计划能执行吗？资源够吗？里程碑是什么？
  太阴：□ 有执行计划  □ 有里程碑  □ 有评估指标
```

## 入口四：分析洞察

### 适用场景

```
你的 Skill 核心是：
  第一层：统计（这件事是什么）
  第二层：洞察（这件事为什么是这样）

判断：用户问的是"有多少"还是"为什么"？
  先问有多少 → 分析洞察 ✓
  只问有什么 → 可能是采集/清洗，跳入口一
```

### 分析洞察特有的"叠"问题

```
统计层（太阳+少阳）：
  太阳工具：频率统计、共现分析、趋势工具、对比工具
  少阳顺序：Step1频率 → Step2共现 → Step3趋势/对比

洞察层（太阳+少阳）：
  太阳工具：根因挖掘、机会识别、策略生成
  少阳顺序：Step4从统计结论 → 提出假设
            Step5验证假设 → 找证据
            Step6形成策略 → 输出建议

核心：统计层的结论 → 洞察层的起点
  不是两个独立 Skill，而是两层架构一个 Skill
```

### 分析洞察特有的"叠"问题（续）

```
洞察层的少阴（判断）比其他类型更重要：

判断H1：统计结论可信吗？
        → 样本量 >= ___ 条
        → 数据来源可追溯

判断H2：洞察有证据吗？
        → 每个洞察 >= 3 条数据支撑
        → 没有反例或反例已解释

判断H3：洞察可行动吗？
        → 不是泛泛而谈
        → 有具体建议/方向

判断H4：假设够大胆吗？
        → 不是显而易见的事实
        → 有洞察增量（用户不知道的）
```

### 分析洞察四象限（洞察层核心工具）

```
            高满意度
                 │
      竞品强     │      保持优势
      （学习）    │      （强化）
                 │
───────┬─────────┼─────────────── 高关注度
                 │
      低优先级   │      立即行动
      （忽略）    │      （机会）
                 │
            低满意度

高关注 + 低满意 = 立即做 🔥🔥🔥
高关注 + 高满意 = 保持 ⭐
低关注 + 高满意 = 传播
低关注 + 低满意 = 暂缓
```

### 分析洞察推导模板

```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
意图（太极）：___的分析洞察

【分】
阳面：统计层（___的频率/共现/趋势）
     洞察层（为什么___/机会在哪/策略建议）
阴面：✗ 不修改原始统计结果
     ✗ 不编造洞察
     ✗ 不跳过统计直接给洞察

【叠】
太阳：统计工具（频率/共现/趋势/对比）+ 洞察工具（根因/机会/策略）
少阳：
  统计层：Step1→频率统计  Step2→共现分析  Step3→趋势/对比
  洞察层：Step4→统计结论→假设  Step5→验证→证据  Step6→策略→建议
少阴：
  统计判断：样本量>=___  数据可追溯
  洞察判断：每个洞察>=3条  无反例  有洞察增量
  四象限：高关注+低满意→立即做，高关注+高满意→保持...
太阴：□ 统计结果可信  □ 洞察有证据  □ 可行动

【展】
乾（输入）：清洗后的结构化数据 + 分析维度
坤（输出）：统计报告（JSON）+ 洞察报告（Markdown）
巽（流向）：流向 voc-insight 或直接给人类用户
坎（异常）：样本不足→停止  洞察无证据→标记待验证
离（标准）：统计+洞察双报告，每个洞察有证据
艮（边界）：不改数据/不编洞察/不跳统计
兑（握手）：统计层输出→洞察层输入，洞察层→人类/下游

【衡】
金→木：统计+洞察双工具链覆盖6步骤 ✓
木→火：统计步骤支撑洞察步骤 ✓
火→水：洞察判断决定流向（给人类/给下游）✓
水→土：流向确定位置（分析链路第3-4步）✓
土→金：位置→需要统计+推理双工具 ✓

【验】
断点：统计结论不可信→回头补样本量
断点：洞察无证据→回头补假设验证

【成】
分✓ 叠✓ 展✓ 衡✓ 验✓
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

```
你的 Skill 有没有上下游？
│
├─ 有，在VOC链路
│    → 入口一（链路节点）
│    → 重点：乾/坤对上游下游
│
├─ 有，在战略链路
│    → 入口三（战略链路）
│    → 重点：L1/L2/L3层级
│
├─ 有，但不进链路
│    → 入口二（独立领域）
│    → 重点：定义自己的握手
│
```
